# Content Factory V1 Risk Register

**Scale:** Probability `Low/Medium/High`; Impact `Low/Medium/Critical`.  
**Release-blocking:** “Yes” blocks collection, package approval, deployment, or publication while open. Risk acceptance must be recorded by the Human Owner; legal/source ambiguity is not waivable by AI.

| ID | Description | Probability | Impact | Detection method | Prevention | Mitigation | Owner | Release-blocking |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| R-01 | Legal/source compliance: collection, retention, quotation, or publication is not permitted | Medium | Critical | Source policy/terms/rights checklist; owner evidence; takedown signal | Register only attributable sources; approve method and rights before collection; no bypass | Stop jobs, quarantine affected items/packages, preserve audit, seek owner/legal decision, remove from target if directed | Human Owner / compliance | **Yes** |
| R-02 | Incorrect property information is normalized or published | Medium | Critical | Field-to-evidence audit; validation; human fact review; source conflict report | Required evidence per field; deterministic rules; no fabrication | Reject/correct as new version, reopen review, supersede/rollback package | Human Reviewer / Publisher | **Yes** for affected record |
| R-03 | Outdated listing/project information appears current | High | Critical | Source dates, “as of” checks, hash refresh comparison, freshness report | Capture published/updated/collected dates; manual refresh policy; label time-sensitive facts | Hold stale fields, recollect approved source, create version, re-review, rollback if published | Human Reviewer | **Yes** for time-sensitive fields |
| R-04 | Duplicate project/property records fragment identity or double-publish | Medium | Medium | Hash/URL/external-ID checks; deterministic name+developer+geography groups | Idempotent ingestion; stable IDs; pre-review and pre-package duplicate gates | Human merge/keep decision; preserve aliases/evidence; rebuild package | Human Reviewer | **Yes** if unresolved |
| R-05 | Price/currency/value basis is wrong or inferred | Medium | Critical | Currency/value/basis/date validation; evidence comparison; high-risk review | Never infer currency; require explicit currency, basis, as-of time, evidence | Mark unknown/reject, correct new version, rollback affected package | Human Reviewer / Publisher | **Yes** |
| R-06 | Evidence is missing, mutable, unlocatable, or hash-mismatched | Low | Critical | Completeness query; payload/restore re-hash; package trace traversal | Evidence-before-parse gate; immutable/versioned store; separate backup | Stop pipeline, quarantine, restore verified copy or recollect only if still approved, re-review | Runtime operator / Human Reviewer | **Yes** |
| R-07 | OCR introduces character/number errors | Medium if used | Critical | OCR confidence; golden scan tests; human side-by-side review | OCR deferred unless unavoidable; local pinned engine; no auto-approval | Quarantine low confidence, manually correct as reviewed version, exclude source/record | Human Reviewer | **Yes** for OCR-derived values |
| R-08 | Parser failure, structure drift, or silent field loss | Medium | Medium | Golden fixtures; required-field deltas; parser failure/empty-content alerts | Source-specific bounded parsers; version pinning; fail closed | Quarantine, revert parser, add approved fixture, reprocess to new version | Technical operator | **Yes** if P0 field/evidence affected |
| R-09 | Publishing package or adapter maps records incorrectly | Medium | Critical | Package contract, deterministic rebuild, staging fixture diff, citation check | One adapter boundary; no direct DB write; exact package approval | Hold publication, restore prior package pointer, fix/re-review new version | Publisher / technical operator | **Yes** |
| R-10 | Rollback fails or prior state cannot be verified | Low | Critical | Timed rehearsal; prior hash/pointer check; audit verification | Preserve immutable prior version; atomic pointer/removal; runbook | Stop publication, isolate package, restore backup, incident review; no release until successful rehearsal | Publisher / runtime operator | **Yes** |
| R-11 | Windows 01 downtime interrupts or loses work | Medium | Medium | Health checks, missed-job alert, restart/fault test, backup status | Durable state, idempotent jobs, bounded scheduler, separate backup | Pause collection, restore state, resume/retry idempotently; run manually only after health restored | Runtime operator | **Yes** if state/evidence integrity uncertain |
| R-12 | Evidence/log/version storage grows beyond capacity | Medium | Medium | Disk threshold, per-job growth metric, backup-size trend | <=100 record cap; temp cleanup; log rotation; bounded versions/retention policy | Stop intake before critical threshold, expand only with approval, archive per policy without deleting required evidence | Runtime operator | **Yes** at safety threshold |
| R-13 | Credential or secret exposure in code, config, logs, evidence, or package | Low | Critical | Secret scan; canary; access audit; log review | Secret references, dedicated identity, redaction, least privilege, no repo secrets | Stop services, revoke/rotate, preserve incident evidence, assess exposure, reauthorize before restart | Human Owner / security | **Yes** |
| R-14 | Cross-project contamination exposes ERP/unrelated data or services | Low | Critical | Path/service inventory; ACL and network denial tests; data-prefix reconciliation | Dedicated identity/paths/network; explicit Windows exclusions | Stop runtime, isolate host, remove foreign data safely, rotate access, incident review | Human Owner / security | **Yes** |
| R-15 | Scope creep adds sources, records, sites, languages, AI, or scale | High | Medium | Manifest count checks; dependency/inventory drift; sprint gate review | Hard limits in config/tests; change control; deferred list | Reject extra work, restore frozen manifest, create separate post-V1 decision proposal | ChatGPT priority lead / Human Owner | **Yes** when boundary exceeded |
| R-16 | Human review bottleneck delays or encourages weak approval | High | Medium | Queue age, records/hour, rework and SLA metrics | 5–10 projects; one checklist; named reviewer; no auto-approval | Pause intake, reduce pilot set (not below 5 without new decision), add approved reviewer with same controls | Human Owner / Reviewer | No for schedule; **Yes** if review bypassed |
| R-17 | Version/audit records become ambiguous or mutable | Low | Critical | Transition tests; history reconciliation; mutation attempt tests | Immutable decisions/snapshots; stable IDs; monotonic versions | Freeze pipeline, restore state, reconstruct only from verified evidence/audit, re-review affected records | Technical operator / Reviewer | **Yes** |
| R-18 | Backup exists but cannot restore evidence/state | Medium | Critical | Clean restore drill and full sampled hash comparison | Separate encrypted target; backup manifest; scheduled verification | Stop deployment/publication, repair backup, rerun clean restore before proceeding | Runtime operator / Human Owner | **Yes** |
| R-19 | Network policy permits unapproved source or production access | Low | Critical | Firewall export; connection deny tests; runtime connection log | Default deny; endpoint allowlist; no public inbound | Stop jobs, close route, review access logs, invalidate affected run, recollect only after approval | Human Owner / security | **Yes** |
| R-20 | Feature Freeze is breached by code, schema, website, deployment, commit, or push activity | Low | Critical | Git/status diff, migration/deployment audit, endpoint logs | Planning-only controls; production endpoints excluded; explicit task stop | Stop immediately, report breach, revert only under separate owner authorization, reassess Alpha RC | Human Owner | **Yes** |

## Risk review cadence

- Review before G1 source approval, G4 Windows deployment, G5 collection, G6 package/publication, and G7 final decision.
- Any new risk is assigned an ID, owner, detection method, and blocking status before work continues.
- A release-blocking risk closes only with objective evidence linked in the gate record.
- Residual risk and reviewer capacity are part of Sprint 8; passing technical tests does not automatically authorize expansion.
